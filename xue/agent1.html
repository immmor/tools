核心架构落地栈

大脑与规划 (Planning) Core

  • ReAct 框架： 深入理解 Reason + Action 循环，解决幻觉与死循环。
  • 规划拆解： Plan-and-Solve、Chain-of-Thought (CoT) 及 Tree of Thoughts (ToT) 的工程实现。
  • 自省机制： Reflexion 与 Self-Correction，让 Agent 具备自我纠错和重试能力。

记忆系统 (Memory) Storage

  • 短期记忆： 窗口管理、Token 裁剪、基于 LLM 的群聊会话摘要（SummaryMemory）。
  • 长期记忆： RAG 深度整合、Embedding 检索优化、元数据（Metadata）精细化过滤。
  • 混合记忆： 模仿人类构建“缓存-工作内存-向量数据库-关系型图谱”的阶梯式记忆。

工具调用 (Tools & Actions) Execution

  • Function Calling： 动态 Schema 生成，极高并发下的 JSON 解析容错与格式修复。
  • 工具发现： 当工具量过大时（100+），如何利用向量检索进行两阶段动态工具匹配。
  • 沙箱环境： Code Interpreter 落地，基于 Docker 隔离的受限代码执行环境设计。

多模态与多Agent (Multi-Agent) Advanced

  • SOP 流程编排： MetaGPT/LangGraph 模式，基于状态机或工作流（Workflow）的确定性交付。
  • 协同模式： Peer Review（对等评审）、Hierarchy（上下级汇报）在垂直业务中的应用。
  • 通信协议： 多 Agent 异步事件驱动架构（Event-Driven）与消息队列设计。

评估与可观测性 (Eval & Observability) Monitor

  • 离线评估： 构建领域专属测试集，结合 LLM-as-a-Judge 与人工标注，量化任务成功率（SR）与轨迹质量。
  • 在线监控： Trace 全链路追踪（OpenTelemetry），Token 消耗、P95 延迟、工具调用失败率实时大盘。
  • 回归防护： Prompt/模型升级前的 A/B + 影子流量（Shadow Traffic）回放，杜绝"改一处崩全局"。

安全与对齐 (Safety & Alignment) Guard

  • Prompt 注入： PromptShield、指令分层（System vs User）、敏感动作二次确认机制。
  • 越权与数据泄露： 工具粒度 RBAC、PII 脱敏中间件、输出审计日志留存。
  • 价值对齐： Constitutional AI、红队对抗测试（Red Teaming）与拒答策略的工程落地。
大厂面试真题与完美作答
Q1. 如何解决生产环境中 Agent 陷入死循环或调用工具失败的问题？
完美作答： 1. 硬限制： 必须在工程层（如 LangChain/LangGraph）设置 Max_Iterations（通常 3-5 次）和 Max_Execution_Time，超时强制中断。
2. 反思 Prompt： 捕获工具返回的 Error 信息，重新喂给大模型，触发 Self-Correction 机制，让大模型换一种参数或工具重试。
3. 兜底策略： 引入规则引擎或经典内控逻辑（如连续 2 次相同输入则降级为人工或返回固定模板）。
Q2. 多 Agent 协作中，如何避免 Token 爆炸以及由于信息传递导致的“传话游戏”误差？
完美作答： 1. 信息按需隔离： 绝不能把 A 的完整 Context 丢给 B。设计精简的 Message Board（公告板机制） 或 Shared State（全局共享状态），Agent 间只传递结构化的 Result，不传递推理过程。
2. 动态裁剪： 严格实施 Message History 剪枝和动态摘要，长文本采用 RAG 形式按需召回。
3. 强类型约束： 强制要求 Agent 间通信使用 Pydantic / JSON Schema，通过协议降低自然语言的二义性。
Q3. 既然有了大模型，为什么在企业落地时更倾向于用 Graph/Workflow（工作流）而不是全自主 Agent？
完美作答： 这是确定性（Deterministic）与灵活性（Generative）的博弈。商业落地核心是“降本增效”与“可控性”。全自主 Agent（如 AutoGPT）在开放场景很炫酷，但在 B 端业务中幻觉率高、成本不可控。 业界最佳实践是 LangGraph 模式：用图和工作流定死核心 SOP 的骨架（确定性），在节点内部长出 Agent 的触角（灵活性），既保住了业务下限，又提升了上限。
Q4. Agent 的记忆系统如何设计？请讲讲你在工程上的具体实践。
完美作答： 记忆不是简单的“塞进 Prompt”，而是分层管理： 1. 工作记忆（Working Memory）： 当前轮次的上下文窗口，结合 Token 计数器做滑动窗口裁剪，并触发 LLM 摘要压缩历史。
2. 情景记忆（Episodic）： 将历史对话片段向量化存入向量库，按语义相似度召回，而非按时间倒序。
3. 语义记忆（Semantic）： 用户画像、偏好等结构化事实，存入 KG 或关系库，走精确检索。
关键工程点： 写入要做去重与冲突合并，召回要做 MMR 多样性重排，避免"全是相似句"导致的信息坍缩。
Q5. 如何评估一个 Agent 系统的好坏？指标怎么定？
完美作答： 不能只看最终答案，要分三层评估： 1. 任务层： 任务成功率（SR）、完成步数、人工满意度评分（CSAT）。
2. 轨迹层： 工具调用准确率、冗余动作率、是否走最短路径，可用 LLM-as-a-Judge 自动评分。
3. 系统层： P95 延迟、单任务 Token 成本、错误恢复率。
实践建议： 建立 Golden Set 黄金测试集做回归，每次 Prompt/模型变更后跑全量对比，指标劣化则阻断发布。
Q6. 面对 100+ 工具，如何让 Agent 又快又准地选对工具？
完美作答： 全部塞进 Prompt 会 Token 爆炸且选择困难。采用两阶段检索： 1. 粗筛（Retriever）： 将工具的 name + description 向量化，根据用户 query 召回 Top-K（如 K=5）候选工具。
2. 精排（Ranker）： 把候选工具的完整 Schema + 示例喂给 LLM，让其精选并生成调用参数。
3. 缓存层： 高频 query 的工具组合做缓存，命中率可达 60%+，显著降本。

从 0 到 1 学习路径

阶段一：基础筑基 Phase 1

  • LLM 基础： Tokenizer、上下文窗口、Temperature/Top-p 采样原理。
  • Prompt 工程： Few-shot、CoT、结构化输出（JSON Mode）。
  • Function Calling： 手写一个天气查询 Agent，跑通 ReAct 循环。

阶段二：工程进阶 Phase 2

  • RAG 实战： 文档切分、Embedding、向量库（Milvus/Qdrant）部署。
  • 记忆系统： 实现滑动窗口 + 摘要 + 向量召回的三级记忆。
  • LangGraph： 用状态机重构一个客服 Agent，加入人工接管节点。

阶段三：生产落地 Phase 3

  • 可观测性： 接入 LangSmith / Langfuse，建立 Trace 与评估集。
  • 成本治理： 模型路由（小模型兜底 + 大模型精排）、缓存策略。
  • 安全上线： Prompt 注入测试、越权审计、灰度发布机制。

阶段四：高阶突破 Phase 4

  • 多 Agent： 事件驱动架构，实现 Planner-Worker-Reviewer 三角协同。
  • 自进化： Reflexion + 经验池积累，让 Agent 越用越聪明。
  • 前沿跟踪： 持续关注 OpenAI Swarm、Computer Use、MCP 协议演进。
高频追问 & 避坑清单
! 面试官最爱追问的“坑”
追问 1： “你说成功率从 65% 提到 95%，怎么测出来的？”
→ 必须回答出测试集规模（如 500 条标注样本）、评估方式（人工 + LLM-as-a-Judge）、置信区间，否则数字不可信。

追问 2： “Token 成本降低 40%，那延迟变了吗？”
→ 经典 trade-off。诚实回答：通过缓存和路由降低调用次数，P50 延迟略降，但 P95 因复杂路由略升 200ms，整体可接受。

追问 3： “如果大模型 API 挂了，你的 Agent 怎么办？”
→ 多供应商兜底（OpenAI/Claude/通义）、降级到规则引擎、关键节点结果缓存、对用户透明提示"服务降级"。
